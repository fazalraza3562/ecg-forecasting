## Project at a glance

We are building an **explainable deep learning framework that predicts life-threatening ventricular arrhythmias from ECG time-series data *before* they occur**. The framing is forecasting, not classification: given the previous 60 seconds of ECG, output the probability that a ventricular arrhythmia (VT/VF) starts in the next 30 seconds.

This is a master's-level course project for BTH course DV2586/DV2646 (Deep Machine Learning).

## The single most important rule

**The report and code must read as if humans wrote them, not an AI.**

This is non-negotiable. The user has explicitly said they want it to look human-written. That means:

- **No AI tells in the report.** Avoid: *delve, tapestry, landscape of, realm of, pivotal, crucial role, multifaceted, nuanced understanding, holistic approach, paradigm shift, leverage (as verb), in conclusion, it is important to note that, furthermore it is worth noting, comprehensive, robust, and scalable*.
- **Vary sentence length aggressively.** Mix 6-word sentences with 30-word ones. Real students do not write evenly.
- **Use first-person plural consistently** ("we trained", "we observed", "we found").
- **Include one or two honest acknowledgments of limitations or surprising negative results** somewhere in the report. Real research has dead ends.
- **Use specific, slightly messy numbers** like "0.847" not "approximately 85%".
- **No bullet points inside report prose.** IEEE conference papers are written in prose. Bullets only appear in code, the contributions list of the intro, and maybe in the appendix.
- **Use em dashes sparingly.** Maybe once or twice per section, not in every paragraph.
- **Cite real papers.** Don't fabricate references. The `.bib` file should only contain papers that actually exist.
- **Some paragraphs should be short.** Three lines is fine. Not every paragraph needs to be a perfect five-sentence block.
- **Don't write "Generated by Claude" anywhere.** This is the group's work product.

In code comments, the same principle applies but with a lighter touch: write comments that a real engineer would write — explaining *why*, not *what*, and only where the code is non-obvious.

## Hard constraints (do not violate)

1. **Patient-level splitting.** Training and test sets must contain *disjoint patients*. A record from patient X cannot appear in both train and test. Violating this inflates results and is the first thing a reviewer would catch. Write a test (`tests/test_patient_split.py`) that enforces it and keep it green.

2. **No leakage from the forecast horizon.** The 60-second input window must end *before* the 30-second prediction horizon begins. No sample from inside the horizon may be visible to the model. Write a test (`tests/test_windows.py`) for this invariant too.

3. **Reproducibility.** Every training script must accept a `--seed` argument and call a shared `set_seed()` helper. Configs live in `config/*.yaml`. No hardcoded paths.

4. **Don't invent results.** Every table and figure in the report must be produced by code that is actually in the repo and actually runs. If `run_all_experiments.sh` produces `results/metrics/sddb_test.csv`, that CSV is what the report quotes. Numbers in the report are real numbers from real runs.

5. **One framework.** PyTorch only. Do not mix PyTorch and TensorFlow.

## How to think about the task

The naïve framing — "classify this ECG beat as normal or arrhythmic" — has been solved many times. What makes this project worth doing is the forecasting framing:

```
input window:        [t-60s ─────────── t]
prediction horizon:                       [t ───── t+30s]   ← will VT/VF start here?
```

The label for a window is `1` if a VT or VF onset annotation falls in the 30-second horizon, `0` otherwise. The model never sees anything inside the horizon. This is the core idea and worth re-reading.

Generate windows with a stride (e.g. 5 seconds) so each recording produces many examples — but make sure every window from a given patient goes into a single split.

## Coding style rules

- Python 3.10+, type hints on every public function.
- PyTorch only.
- Every model is an `nn.Module` whose `forward(x)` returns a single logit per window: shape `(B,)`.
- Training is done by `src/training/train.py`, which takes `--model {baseline_lstm,cnn_lstm_attention,transformer,resnet1d,inception1d,tcn}` and a config path. The trainer is model-agnostic — it doesn't import any specific model.
- Use `logging` (stdlib), not `print`.
- Checkpoints save to `runs/<model>/<YYYYMMDD-HHMMSS>/best.pt` along with `meta.json` recording: git SHA, config snapshot, seed, dataset hash.
- No global state. No `import *`. No magic numbers — everything important comes from config.
- Keep functions short. If a function is over 50 lines, split it.
- Docstrings: one-liner summary, then a blank line, then a paragraph. Google or NumPy style, pick one and stick with it.

## Evaluation discipline

Report these metrics for every model, on both the held-out SDDB test patients **and** on the cross-dataset MIT-BIH / INCART test:

- AUROC
- AUPRC (the class imbalance matters; always report AUPRC alongside AUROC)
- Sensitivity at 95% specificity (a clinically meaningful operating point)
- F1 at the optimal threshold on validation
- False positive rate per hour of monitoring (clinical metric)
- Median early-warning lead time in seconds, with IQR

For the proposed model only, also include:

- Ablation: drop CNN, drop LSTM, drop attention — three rows in a table.
- Per-patient AUROC distribution boxplot (shows generalization spread).
- Failure-case analysis: two or three representative false negatives with attention overlay.
